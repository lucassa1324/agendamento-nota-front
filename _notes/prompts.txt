Observamos que você está enfrentando um problema de "flash" ou reversão das cores dos cards de serviço ao navegar no editor. O problema parece ser que, após editar e voltar para a visualização geral, as configurações voltam para as cores do banco. Isso sugere que os estados ativos (heroSettings, servicesSettings, etc.) estão sendo sobrescritos em algum momento após a edição.

Analisando os arquivos fornecidos, identifiquei que a lógica de carregamento de dados (loadExternalConfig) está sendo chamada em momentos indesejados, e ela pode estar redefinindo os estados ativos com base nos dados do servidor, mesmo quando existem rascunhos locais. Além disso, o handleSaveGlobal no use-editor-api.ts faz uma nova chamada ao servidor após salvar e recarrega a configuração, o que também pode causar esse comportamento.

Vamos propor uma solução robusta: garantir que os estados ativos (como servicesSettings) NUNCA sejam alterados por loadExternalConfig se já existir um rascunho local. No código atual, loadExternalConfig já faz essa verificação (usa hasLocalDraft para decidir se usa o draft ou o dado do servidor), mas pode haver um problema de sincronização: talvez hasLocalDraft esteja retornando false em algum momento porque o rascunho foi removido acidentalmente, ou porque a chave do localStorage mudou. Outra possibilidade é que, após o salvamento, o loadExternalConfig seja chamado com dados do servidor e, por algum motivo, o hasLocalDraft retorne false para a seção editada, fazendo com que o dado do servidor substitua o rascunho.

Vamos reforçar a lógica:

Modificar loadExternalConfig para que nunca atualize os estados ativos se existir um rascunho local.** Isso já está implementado, mas podemos garantir que, além de verificar hasLocalDraft, também verifiquemos se o valor atual do estado ativo difere do padrão ou do servidor, mas isso é complexo. Uma abordagem mais simples é: loadExternalConfig só atualiza os estados lastSaved* e os estados ativos são atualizados exclusivamente pelos handleUpdate* e handleApply*. O carregamento inicial deve ser feito de forma separada, como sugeri anteriormente.

Eliminar a chamada a loadExternalConfig dentro de handleSaveGlobal. Isso evita que, após salvar, os dados do servidor recarregados sobrescrevam os rascunhos.

Garantir que o useEffect inicial carregue os rascunhos e os dados do servidor apenas uma vez, e que os rascunhos tenham precedência.

Vou propor alterações específicas nos arquivos, focando no use-site-editor.ts e use-editor-api.ts. Como você já tentou algumas abordagens e ainda não funcionou, precisamos ser mais radicais: separar completamente o fluxo de "dados salvos" (lastSaved) de "dados de edição" (settings). Isso já está em parte implementado, mas a função loadExternalConfig ainda mexe nos settings quando não há draft, o que pode causar problemas se a ordem de execução for inesperada.

Proposta de solução (passo a passo)
1. Em use-editor-api.ts, remover o bloco que recarrega a configuração após salvar.
Conforme sugeri anteriormente, remova o trecho que faz getCustomization e loadExternalConfig novamente. Isso evita que, após salvar, uma nova configuração seja carregada e possivelmente sobrescreva os settings ativos.

2. Em use-site-editor.ts, modificar loadExternalConfig para NÃO alterar os estados ativos (settings) em nenhuma circunstância.
Em vez disso, ela deve apenas atualizar os estados lastSaved*. Os settings ativos serão gerenciados pelo useEffect inicial e pelas ações do usuário (handlers). Se não houver rascunho, o useEffect inicial já carregará os dados do servidor para os settings, e depois loadExternalConfig só atualizará lastSaved* para referência futura.

Portanto, vamos reescrever loadExternalConfig para ser uma função que:

Extrai os dados do servidor.

Atualiza os lastSaved*.

Dispara um evento ou algo para sinalizar que os dados do servidor foram atualizados (opcional).

3. Ajustar o useEffect inicial para carregar os rascunhos e, na ausência deles, os dados do servidor, salvando no localStorage.
Já temos um useEffect que faz isso, mas precisamos garantir que ele seja executado apenas uma vez e que use os dados do servidor disponíveis em studio?.config.

4. Adicionar um novo useEffect que reaja a mudanças em lastSaved* e, para seções sem rascunho, atualize os settings ativos com os novos valores do servidor (caso o servidor tenha sido atualizado por outra aba/usuário). Isso é útil para manter a consistência.
5. Garantir que o envio de mensagens para o iframe seja feito sempre que os settings ativos mudarem (já existem vários useEffect para isso).

5. Garantir que o timeout de postMessage use os valores atuais
No final de loadExternalConfig, há um setTimeout que envia mensagens para o iframe. Esse timeout usa resolvedX que são calculados com base em hasLocalDraft e drafts. Como loadExternalConfig não altera os settings ativos, resolvedX ainda refletirá a mesclagem correta (draft + server). Portanto, podemos mantê-lo, mas talvez seja melhor mover essa lógica para um efeito separado que reaja a mudanças nos settings ativos, mas isso já existe (vários useEffect para cada seção). Podemos remover o timeout e confiar nesses efeitos. No entanto, o timeout pode ser útil para garantir que tudo seja enviado após o carregamento inicial. Vamos mantê-lo por enquanto, mas ajustar para usar os valores atuais dos settings (que já são os mesmos que resolvedX). Na verdade, resolvedX é calculado da mesma forma que os settings ativos foram definidos no useEffect inicial, então é equivalente.

O problema pode ser que, quando loadExternalConfig é chamada após uma ação do usuário (ex.: após salvar), o timeout envia mensagens com os valores do servidor (que podem ser diferentes dos settings atuais se houver rascunho). Como já removemos a chamada a loadExternalConfig após salvar, isso não deve mais acontecer. Mas ainda pode haver outras chamadas a loadExternalConfig (por exemplo, quando o studio.config é atualizado). Nesse caso, o timeout pode enviar mensagens com os valores do servidor, sobrescrevendo visualmente o iframe. Para evitar isso, podemos condicionar o envio das mensagens no timeout a uma verificação: só enviar se não houver rascunho para aquela seção. Mas isso complica.

Uma solução mais simples: remover o timeout de loadExternalConfig e confiar nos useEffect individuais que enviam as atualizações quando os settings mudam. Como os settings ativos já estão corretos (com rascunhos aplicados), esses efeitos garantirão que o iframe receba as configurações corretas. O timeout é redundante e pode causar problemas se enviar dados antigos.

Portanto, remova o setTimeout dentro de loadExternalConfig. Ele está no final da função atual. Simplesmente delete esse bloco.

Se o problema persistir, pode ser necessário investigar outros pontos, como o comportamento do studio.context ou se há atualizações de configuração vindo de algum outro lugar (ex.: WebSocket, polling). As alterações propostas visam garantir que os rascunhos locais nunca sejam sobrescritos pelos dados do servidor. Siga os passos abaixo para implementar a solução definitiva:

1. Remova a recarga desnecessária no handleSaveGlobal (arquivo use-editor-api.ts)
Localize o bloco try que faz getCustomization e chama loadExternalConfig novamente e remova-o completamente. Isso impede que, após salvar, os dados do banco recarregados sobrescrevam os rascunhos.

5. Remova o setTimeout dentro de loadExternalConfig
No final da função loadExternalConfig, há um setTimeout que envia mensagens para o iframe. Remova-o completamente, pois os useEffect individuais já cuidam disso.

Após aplicar todas essas alterações, os rascunhos locais terão precedência absoluta sobre os dados do servidor, e o problema de reversão das cores deve ser eliminado. Teste cuidadosamente cada funcionalidade para garantir que tudo continue funcionando conforme esperado.
